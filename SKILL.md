# SKILL.md — 반복 업무 표준 절차 (정철)

## 1. 시험 과목 문제 우선 풀이법 (제조공학 등 전공 시험)
**트리거**: "~단원 시험 준비", "~챕터 재시험", "문제 풀어줘" 요청 시

절차:
1. 이론 설명을 먼저 하지 않는다. 바로 기출/유사 문제를 제시하거나 풀게 한다.
2. 틀리거나 막힌 지점만 짚어서 그 개념만 콕 집어 설명한다 (전체 챕터 이론 X).
3. 설명할 때 가능한 경우에만 SprayLab/야구 통계 비유를 쓴다 (억지 금지, 규칙 2 참고).
4. 마지막에 유사 문제 1~2개로 재확인한다.
5. 목표는 "합격 가능 점수"이지 "완벽 이해"가 아님을 항상 기준으로 삼는다.

## 2. SprayLab 기능 개발/디버깅
**트리거**: SprayLab 코드, Supabase 스키마, GitHub Actions 관련 요청

절차:
1. 현재 배포 구조(GitHub Pages + Supabase)를 벗어나는 제안 시 이유를 먼저 설명한다.
2. UI 버그는 가능하면 재현 조건부터 확인 후 수정한다.
3. 신규 통계 지표(wOBA, BABIP 등) 추가 시, 계산식 출처를 명시하고 KBO 공식 정의와 다르면 표시한다.
4. 마케팅/SEO 관련 변경은 브랜드 톤(과장 없이, 신뢰도 우선)을 지킨다.

## 3. KBO 세이버메트릭스 분석/콘텐츠 제작
**트리거**: 통계 분석, 릴스 스크립트, 카드뉴스, 발표 자료 요청

절차:
1. 최신 KBO 기록/순위/스탯이 필요하면 반드시 검색으로 확인 시점을 명시한다 (기억에서 단정하지 않음).
2. 통계적 주장(예: ABS 도입 효과)은 검정 방법(t-test 등)과 표본을 같이 명시한다.
3. 콘텐츠용으로 각색할 때는 후킹 문장 1개 + 근거 데이터 1개를 세트로 묶는다.
4. 결과물 끝에 "왜 이 구성인지" 1줄 근거를 남긴다.

## 4. 경량 모델(Haiku 등) 사용 시 강제 체크리스트
경량 모델에서 품질이 떨어지는 걸 막기 위한 보완 절차 (자기비판 반영):
1. 작업 시작 전 "검증 필요 항목" 목록을 먼저 뽑는다 (최신 정보/수치/정책).
2. 각 항목마다 검색 가능하면 검색, 불가능하면 "미검증" 표시를 답변에 남긴다.
3. 출력 전 위 CLAUDE.md 규칙 6개를 셀프 체크리스트로 훑는다.

## 5. 모델 전환 시 컨텍스트 이어받기
Sonnet ↔ Opus/Fable ↔ Haiku 전환 시:
1. 직전 대화의 "역할"(코드/학습/콘텐츠)과 "목표"를 한 줄로 요약해 새 세션 상단에 붙인다.
2. CLAUDE.md 6개 규칙이 자동 적용 안 되는 환경(예: 순수 챗봇 UI)에서는 범용 커스텀 인스트럭션(진단 리포트 4번 참고)을 직접 시스템 프롬프트에 넣는다.
